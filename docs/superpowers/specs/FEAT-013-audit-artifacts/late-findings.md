## MF-063 — CONFIRMED — high — bug — Verdict frontmatter enum drift: `APPROVED` token not in VALID_VERDICTS blocks the DONE-gate
- **class**: bug (enum drift — same class as MF-001, different field)
- **evidence**: `scripts/lib/structured-state.sh` `VALID_VERDICTS` recognizes the literal token `APPROVE` only; `read_verdict` returns INVALID for `verdict: APPROVED` (past participle). The TASK-010 review gate persisted all four verdicts as `verdict: APPROVED`, so `task-state-guard.sh`'s DONE-gate reported "Review does not contain APPROVED verdict" for all four despite genuine approvals — the fail-loud INVALID path never reached the legacy-regex fallback that would have matched. Live-reproduced this run: TASK-010 IN_REVIEW→DONE was blocked until the orchestrator normalized the four `nazgul/reviews/TASK-010/*.md` tokens to canonical `APPROVE`. TASK-001..009 gates all used `APPROVE` and passed cleanly — so the producer side (review-gate/reviewer output) is not pinned to a canonical token, and a drift to `APPROVED` silently wedges the gate.
- **failure scenario**: any review gate (or reviewer) that emits `verdict: APPROVED` instead of `APPROVE` produces four genuine approvals that the DONE-gate cannot read, wedging the task at IN_REVIEW with a misleading "no APPROVED verdict" message — indistinguishable to an operator from an actual rejection. In an unattended AFK/YOLO loop this stalls the task until max-iteration/consecutive-failure gates trip.
- **recommendation**: (1) add `APPROVED` as an accepted alias in `VALID_VERDICTS` (mirrors the MF-001 fix for status `APPROVED`), OR (2) pin the review-gate/reviewer verdict emission to the canonical `APPROVE` token with a schema check. Prefer BOTH — accept-alias on the consumer side (defensive) + pin the producer. Bundle with MF-001's fix as one "enum-drift: accept the documented alias everywhere the guard reads a controlled vocabulary" work item.
- **cross-links**: MF-001 (identical bug class, status field); MF-002 (INVALID is a silent black hole in the consumer); dimension 1, dimension 2.
- **provenance**: found by the TASK-010 verifier while walking its own manifest to DONE; it correctly refused to edit persisted review files or plugin source (both out of its read-only scope) and escalated — good behavior, the FEAT-006 forged-manifest lesson holding.
