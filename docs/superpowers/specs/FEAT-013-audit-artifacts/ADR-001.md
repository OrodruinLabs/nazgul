# ADR-001 (FEAT-013): Parallel Dimensional Sweep with Adversarial Verification for a Read-Only, Report-Only Audit

## Status
ACCEPTED — 2026-07-22. Scoped to FEAT-013 (360 Reliability Audit). Analysis-only: NO release, NO
version bump, NO schema change. Supersedes the FEAT-012 ADR-001 (superseded by objective rotation,
not by contradiction).

## Context
The operator reports persistent flakiness across the plugin (loop engine, review board, guards,
parallel/heartbeat/connectors). Reliability debt is scattered across `nazgul/improvements.md`,
session memory, and undiagnosed runtime evidence. Prior handling has been whack-a-mole, incident by
incident. This objective performs one comprehensive audit; the deliverables are documents
(findings register, structural critique, fix roadmap), not code changes. Five decisions shape how
the audit runs. The approved design is
`docs/superpowers/specs/2026-07-22-nazgul-360-audit-design.md`.

## Decision 1: Parallel dimensional sweep, not sequential deep-dive (CHOSEN: parallel)
- **Chosen**: 8 read-only agents run in parallel, each owning one dimension with an explicit file
  scope and mandatory anchor list; the orchestrator keeps only synthesis in-context.
- **Alternative — one sequential deep-dive by the orchestrator**: a single context walking ~181
  files across 8 concerns would exhaust the context window long before dimension 8, degrading late
  dimensions to skimming and losing early evidence to compaction. Rejected: **context exhaustion**
  is the primary failure mode of a whole-plugin audit; per-dimension agents each get a fresh
  context sized to their scope.
- **Alternative — fewer, broader agents (e.g., 3)**: still overloads each context and blurs
  accountability for anchors. Rejected.
- **Consequence**: cross-dimension duplicates are expected and handled in a dedup phase, where
  independent rediscovery is treated as a severity signal rather than noise.

## Decision 2: Adversarial verification bar — critical/high + wave-1 mediums (CHOSEN)
- **Chosen**: every critical and high finding, plus any medium slated for roadmap wave 1, gets a
  fresh skeptic agent whose sole job is refutation with `file:line` proof. Survivors = CONFIRMED;
  skeptic-killed findings are demoted or dropped; everything unverified ships labeled PLAUSIBLE and
  is never silently upgraded.
- **Alternative — verify everything**: doubles agent cost for low/medium findings that don't drive
  the roadmap's head. Rejected as cost without decision value.
- **Alternative — verify nothing / trust the sweep**: LLM audits produce confident phantom
  findings; the FEAT-006 forged-manifest episode and the stale-diff phantom-findings incident prove
  self-reported findings need hostile checking before the operator acts on them. Rejected.
- **Consequence**: the report's CONFIRMED label is a hard guarantee (independent skeptic survived),
  which is exactly what acceptance criterion 2 requires.

## Decision 3: Strictly read-only — no fixes applied, even trivial ones (CHOSEN)
- **Chosen**: the audit never modifies plugin source. Findings — including one-line obvious
  defects — go in the report only. No version bump, no CHANGELOG, no release.
- **Alternative — fix trivia opportunistically while auditing**: mixes measurement with mutation:
  the audited surface shifts mid-audit, dimension scopes desync from disk, "trivial" fixes skip the
  review board, and the diff pollutes an analysis-only objective. Rejected.
- **Consequence**: remediation happens later via the fix roadmap's chartered waves, each with its
  own review cycle; the audit's evidence stays reproducible against v2.17.3.

## Decision 4: Final report stays local and uncommitted (CHOSEN)
- **Chosen**: `docs/superpowers/specs/2026-07-22-nazgul-360-audit.md` is written locally and NOT
  committed. Working artifacts live under `nazgul/context/objectives/FEAT-013/` (runtime state,
  never committed). No remote publishing of any deliverable (no Artifact tool, no gists).
- **Alternative — commit the report to main or a branch**: violates the operator's standing rule
  (specs stay local/uncommitted; work ships later via branch + PR) and would bake unverified
  self-criticism into repo history before the operator triages it. Rejected.
- **Consequence**: the report's lifecycle is operator-owned; waves that get green-lit ship through
  the normal branch + PR flow later.

## Decision 5: Review board reviews AUDIT ARTIFACTS, not code diffs (CHOSEN)
- **Chosen**: for this objective, the reviewer board's subject is the audit outputs themselves —
  evidence quality (concrete `file:line` that actually says what's claimed), refutability (findings
  stated falsifiably), and coverage (all anchors root-caused or cleared, scope complete, bounded
  coverage disclosed).
- **Alternative — skip the review gate (no code, nothing to review)**: the report drives the entire
  future fix roadmap; an unreviewed report with phantom or unevidenced findings is worse than no
  report. Rule 5 ("never skip the review gate") applies to the deliverable, whatever its medium.
  Rejected.
- **Alternative — run the standard diff-based review**: there is no diff; a diff-scoped board would
  vacuously approve. Rejected as vacuous.
- **Consequence**: reviewers evaluate documents in `nazgul/context/objectives/FEAT-013/` and the
  final report; verdict semantics (APPROVE / CHANGES_REQUESTED / UNVERIFIED, critical reviewers
  fail closed) are unchanged.

## Consequences (overall)
- Positive: whole-plugin coverage without context collapse; a findings register whose CONFIRMED
  entries are adversarially guaranteed; one consolidated fix queue absorbing `improvements.md`; zero
  risk to the shipped plugin during the audit.
- Negative / accepted: parallel sweep costs more agent dispatches than a single pass; PLAUSIBLE
  findings carry residual uncertainty by design; the un-committed report must not be lost before
  operator triage (it is the single synthesis artifact).
