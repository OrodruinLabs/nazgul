# Nazgul Plugin — 360 Reliability Audit

**Date:** 2026-07-22 · **Plugin version audited:** v2.17.3 · **Objective:** FEAT-013
**Status:** Final report (local, uncommitted per standing spec-handling rule)
**Method:** 8 parallel read-only dimensional sweeps → dedup/merge → adversarial verification → this synthesis. All 10 upstream tasks passed their own review boards. Design: `docs/superpowers/specs/2026-07-22-nazgul-360-audit-design.md`. Working artifacts: `nazgul/context/objectives/FEAT-013/`.

---

## 1. Executive TLDR

The plugin has **63 distinct reliability findings** (62 in the merged register + 1 found live during the audit — MF-063, high): **10 critical, 19 high, 18 medium, 16 low**. Every critical and high finding was independently verified against live source by a refutation-oriented skeptic **and** re-derived from scratch by at least one review-board reviewer — **0 were refuted**, one was reclassified (MF-027, see below), one had its mechanism narrowed (MF-014).

**The one-sentence finding:** the framework's *enforcement* and its *self-reporting* both claim more than they deliver — several guards RULES.md marks `[enforced]` do not actually run, and the loop's own status/telemetry can silently go stale — and the reason this shipped is that **the test suite exercises input shapes and code paths that production no longer uses**, so the suite stays green while real code is broken.

**The five highest-impact confirmed defects:**

1. **The git-level safety net never installs (MF-034, critical).** `install_git_hooks` is only reachable through `worktree-utils.sh`, which *no production code path sources* — `skills/start/SKILL.md` re-implements branch setup as prose instead. Proven live on this repo: `guards.git_hooks: true` and an active objective, but `core.hooksPath` is the OS default and `nazgul/.githooks/` doesn't exist. The FEAT-010 base-branch and pre-merge-verdict guards — the flagship "enforce at the layer that knows the truth" work — protect nothing today.

2. **Bash-mediated writes bypass the entire state machine (MF-022, critical).** `task-state-guard.sh` is wired only to `Write|Edit|MultiEdit`, never `Bash`. A forged `status: DONE` manifest can be placed with one `mv` (or `python3 -c`, `perl -i`, …) and never touches the guard, the evidence gates, or the review-gate check.

3. **The `APPROVED` status is unspellable (MF-001 → MF-002/004/005, critical).** `VALID_STATUSES` omits `APPROVED`, so any task at that state reads as `INVALID` — and `INVALID` is a *silent black hole* in all four of the loop's duplicated task-counting loops (no bucket, no diagnostic). In YOLO+task-PR mode this wedges dependency promotion and makes loop completion unreachable, with no error identifying why.

4. **Group/feature review granularity can never reach DONE (MF-013, critical).** The DONE-gate and its reactive safety net both key review evidence by `TASK-ID` only, while group mode writes evidence to `GROUP-<n>/`. No bridge exists — so the template's own default granularity path is structurally non-completable.

5. **Reviewer verdict-capture stalls are the single highest-recurrence defect (MF-014, critical).** Reproduced across FEAT-009/010/012 and **dozens of times in this very audit's own review gates**. Multi-causal (verified): `review-gate.md` never pins synchronous dispatch, reviewers hit a 12-turn budget cap, haiku-tier format non-adherence, and this run's Agent-Teams dispatch is async-by-design.

**Trust note on this report:** verification was performed by one self-skeptic across all 38 queue findings rather than 38 independently-briefed agents (the manifest's literal protocol). This is disclosed, and heavily mitigated — ~24/38 were empirically reproduced (two fired against the verifier's *own* commands mid-session), and four review-board reviewers independently re-derived a 6–10 finding sample from scratch and corroborated every one. The security reviewer (critical) re-verified the highest-stakes findings from live source (MF-001, MF-002, MF-022, MF-027, MF-014 by its persisted account) and concluded the register is safe to act on. Treat the individual verdicts as trustworthy; treat "37/38 confirmed" as "confirmed by one rigorous self-skeptic + spot-corroborated by four reviewers," not "by 38 independent agents."

## 2. Root-cause clusters

The 63 findings collapse into **7 diseases**. Fixing by cluster is far cheaper than fixing by finding, because within each cluster one structural change closes many symptoms.

### Cluster A — Controlled-vocabulary drift (enum desync + silent-INVALID)
`VALID_STATUSES`, the guard's own status regex, `VALID_VERDICTS`, and various doc comments each maintain the state/verdict vocabulary *independently*, and they've drifted. Worse, an off-vocabulary value resolves to `INVALID`/`INVALID`-verdict which is treated as a **silent black hole** rather than a loud error.
→ MF-001, MF-002, MF-004, MF-005, MF-010, MF-011, **MF-063** (verdict-token twin, found live). Fixing the vocabulary *and* making INVALID loud closes six findings.

### Cluster B — Copy-pasted logic that drifts
The same logic exists in 2–4 hand-maintained copies with no shared helper, so a fix or feature added to one copy silently skips the others.
→ MF-009 (task-counting ×4 — the direct cause of MF-002's four-place gap and MF-008's one-of-four gap), MF-033 (two bespoke shell tokenizers — the engine behind recurring quoting bugs MF-027/MF-028), MF-018 (classification policy ×2), MF-003 (awk coupled to one exact plan.md format).

### Cluster C — Enforcement theater (guards that don't run, or don't enforce what RULES.md claims)
The most security-relevant cluster. Guards are marked `[enforced]` but check the wrong tool, the wrong field name, an unset env var, an unparseable format, or aren't wired at all — or fail *open*.
→ MF-034+MF-035 (git-hooks never install / worktree escape), MF-022 (Bash bypass), MF-023 (prompt-guard reads a nonexistent env var → dead), MF-024 (File Scope guard queries a field planner never emits → dead), MF-026 (commit-SHA "gate" is a regex, no `git cat-file`), MF-027 (over-block) / MF-028 (order-dependent bypass), MF-053 (fail-open on corrupt config), MF-047 (Teammate Contract layers 1–2 advisory), MF-051 (`safety.block_destructive_commands` is a dead key — sets nothing), **MF-038** (critical — the GitHub connector's documented two-way sync is silently pull-only: the push half's map can never resolve a real local id, so `push_status`/`push_pr` are unconditional no-ops with no error).

### Cluster D — Review-pipeline reliability & integrity
`review-gate.md`'s dispatch and persistence contract is under-specified: no synchronous pin, no canonical verdict filename, no trust boundary on inbound messages, no single classification source.
→ MF-014 (stalls, multi-causal), MF-013 (group evidence keying), MF-015 (coverage inferred not verified), MF-016 (emit-event drops verdict events), MF-058 (4 incompatible verdict-file schemes — one blocked a DONE transition live), MF-059 (injected coordination indistinguishable from attack — 4 occurrences this run), MF-017/MF-019/MF-020 (naming/dead-key/sprawl), **MF-063** (verdict-token drift, also Cluster A).

### Cluster E — Docs/config drift & dead config
No forcing function keeps docs and config in sync with code; migrations delete keys but not the docs that describe them, and dead keys give operators a false sense of control.
→ MF-049 (whole stale Execution Engine section citing deleted files), MF-048 (`.bak` sprawl + secrets in backups), MF-050 (schema-stale live config, mixed key trees), MF-051 (dead-key clusters), MF-029 (CLAUDE.md omits 5 live hooks), MF-043/MF-046 (numbering/label drift).

### Cluster F — Observability & recovery gaps
State that's supposed to survive compaction or enable recovery goes stale silently, and sanctioned alternative execution paths bypass the accounting hooks entirely.
→ MF-003 (Recovery Pointer frozen — a no-op against live plan.md), MF-007 (checkpoint schema collision loses richer state), MF-060 (**this run** is invisible to plan.md/telemetry because Agent-Teams dispatch skips the loop's emit hooks), MF-062 (improvements.md: 74 open, 0 ever closed), MF-008 (compact hooks lack granularity awareness), MF-039 (heartbeat TOCTOU), MF-044 (heartbeat silently drops an inbox item on failure), **MF-040** (high — Wave Groups line-format brittleness silently degrades `execution.parallel` to fully sequential when a plan lists one task per bullet instead of comma-joined, with no signal; this very plan.md's Wave 1 hits the pattern).

### Cluster G — The meta-cause: tests don't test production shapes
Why every other cluster shipped green. The suite exercises formats and paths production doesn't use.
→ MF-052 (fixtures write the legacy status format across 259 call sites; production writes only canonical frontmatter — already caused one production-only crash), MF-055 (shellcheck array is hardcoded and stale: 41% of scripts unchecked, including the load-bearing `task-utils.sh`/`structured-state.sh`), MF-053/MF-041/MF-054/MF-056 (fail-open branches with zero coverage), MF-023 (prompt-guard test uses an env var production never sets), MF-057 (skips reported as passes).

The full severity-ranked register — every finding with `file:line` evidence, failure scenario, recommendation, and per-finding verification verdict — lives in **`nazgul/context/objectives/FEAT-013/merged-findings.md`** (62 findings, MF-001…MF-062), with per-finding skeptic records in **`verification-verdicts.md`** and the git-hooks/injection dogfooding detail in **`nazgul/context/objectives/FEAT-013/late-findings.md`** (MF-063 + orchestration meta-observations). Counts: 10 critical, 19 high (MF-063 is high per its source label in `late-findings.md`), 18 medium, 16 low; 0 evidence-less drops; verification tally **36 CONFIRMED + MF-027 reclassified + MF-014 symptom-confirmed/mechanism-partial + MF-042 PLAUSIBLE**, 0 refuted.

Rather than restate all 63 here, the register is the reference; the roadmap below is organized so every confirmed critical/high maps to a wave.

**Two verification refinements the board forced (evidence integrity working):**
- **MF-027 was reclassified** from an implied fail-open framing to a **fail-safe over-block** — the unanchored `rm -rf /` regex blocks legitimate `rm -rf /tmp/…` (it blocked the verifier's own diagnostic command mid-run). It's a usability/availability defect (fix: anchor the regex), **not** a security bypass, and must not be prioritized alongside the genuine bypass MF-022.
- **MF-014's mechanism was narrowed** by a fresh differently-primed skeptic: the stall *symptom* is confirmed and cross-objective, but "the Agent tool defaults to background dispatch" is only one of several causes, and this run's own stalls actually come from a *different* dispatch path (Agent-Teams) than the finding indicts. The fix must be multi-pronged, not just `run_in_background: false`.

## 4. Structural critique (challenge-everything lens)

Beyond individual bugs, four structural patterns are worth a deliberate decision, not just a patch:

1. **The command-string-parsing arms race is unwon and unwinnable in that form.** Two hand-rolled ~120-line quote-aware tokenizers (`pre-tool-guard.sh`, `local-mode-tracking-guard.sh`) independently try to infer intent from shell text, and quoting bugs keep reappearing (MF-027, MF-028) because a fix in one can't propagate to the other. The project already learned this once (FEAT-010 moved base-branch enforcement to a git hook) — but MF-034 shows that git-level move never actually activated. **Recommendation:** finish the migration to enforcement at the layer that observes the resolved action (git hooks for git actions, a PreToolUse guard on the *resolved* `tool_input` for file writes including Bash), and extract one shared tokenizer for whatever advisory parsing remains.

2. **`worktree-utils.sh` is a complete, well-tested, orphaned library.** It's the clearest "the wiring, not the code, is the bug" case in the repo. Either wire it in (the fix for MF-034) or delete it and rebuild the install trigger in `session-context.sh`. Leaving a tested-but-uncalled safety library in place is worse than either.

3. **`review-gate.md` has grown to a ~620-line, 12-step pipeline with out-of-order numbering (Step 3.6)** and is the single point where the most-recurring defects cluster (MF-013/14/15/16/17/58/59/63). It needs a *contract*, not more steps: one canonical verdict-file name, an explicit synchronous-dispatch pin, a stated trust boundary for inbound messages, and one classification source shared with `feedback-aggregator.md`.

4. **Config and docs have no forcing function.** Config schema is at v27 with dead keys from v5 still documented as live (MF-049/051), and `improvements.md` has 74 open items and *zero* closures across three objectives (MF-062). The self-improvement loop *files* findings well but has no mechanism to *retire* them — this audit had to identify 4 already-fixed items (IMP-047/048/056/072) by hand. A closure/retirement step belongs in the post-loop gate.

**Is the plugin overbuilt?** In places, yes — the dead-key clusters (MF-051), the four config-key groups with no consumers, and the 12-step review pipeline are complexity that breeds the drift bugs above. But the core architecture (files-as-memory, state machine, review board, evidence gates) is sound and its *evidence-integrity* mechanisms (recompute-and-compare, UNVERIFIED fail-closed) demonstrably held up under this audit's own adversarial pressure. The problem isn't the design; it's that enforcement and test-realism didn't keep pace with the design's growth.

## 5. Fix roadmap

Dependency-ordered waves, each a charterable Nazgul objective. Wave 1 is foundational and cheap — several one-line fixes with outsized leverage — and, critically, **fixes the test-realism meta-cause first so later waves don't ship green-but-broken again**.

### Wave 1 — Foundations & test-realism (unblocks everything; mostly one-line, highest leverage)
**Charter:** make the loop's core vocabularies single-sourced and loud on error, consolidate the duplicated counting logic, and fix the test fixtures so the rest of the roadmap can be verified.
- **Enum-drift bundle:** MF-001 + MF-063 (add `APPROVED` to `VALID_STATUSES` and `VALID_VERDICTS`; reconcile against `task-state-guard.sh`'s independent lists — one source of truth) + MF-002/MF-011 (add a loud `INVALID` arm to the counting loops) + MF-010 (template comment). **Side-effect fixes:** MF-004, MF-005. **Retires:** none new.
- **Counting consolidation:** MF-009 — extract `count_tasks_and_find_active()` into `task-utils.sh`; carries the INVALID fix once instead of four times; structural fix for MF-002/MF-008.
- **Test-realism (do this or Wave 2–4 regress):** MF-052 (default `create_task_file()` to canonical frontmatter) + MF-055 (replace the hardcoded shellcheck array with a `find` glob).
- **Recovery/observability quick wins:** MF-003 (format-tolerant Recovery Pointer awk + post-write diff) + MF-060 (flag stale summary / wire Agent-Teams path to emit hooks).
- **Acceptance:** a canonical `status: APPROVED` manifest completes a YOLO loop; INVALID emits a diagnostic; the shellcheck test covers all 50 scripts; `nazgul/plan.md` reflects real task state during an Agent-Team run.

### Wave 2 — Guard integrity (the security spine)
**Charter:** make every guard RULES.md marks `[enforced]` actually run and actually enforce; finish the git-level migration.
- **Git-hooks activation (ship together):** MF-034 (wire `install_git_hooks` into the real branch-setup site) + MF-035 (worktree-cwd-safe merge / per-worktree hook scoping) + MF-036 (p4 names + drift detection).
- **State-machine coverage (the "guard coverage gap" bundle):** MF-022 (enforce manifest writes at the resolved-`tool_input` layer incl. Bash) + MF-026 (`git cat-file -e` real SHA verification). Per the security reviewer, ship with a test each.
- **Dead/miswired guards:** MF-023 (prompt-guard → stdin), MF-024 (File Scope → `Files modified`) + MF-025 (shared JSON-array accessor, needed by MF-024).
- **Parallel-dispatch correctness (same JSON/format-parsing family as MF-025):** MF-040 (parse each wave's membership from the `### Wave N` heading + its `- TASK-NNN` bullets rather than requiring same-line comma-grouping, so `execution.parallel` can't silently degrade to sequential).
- **Connector correctness:** MF-038 (thread the picked issue number through to a real local id so the GitHub connector's push half can actually resolve and write back — closes the silent pull-only gap; opt-in/default-off, so lower blast radius but a critical correctness defect for anyone enabling `connectors.github.push`).
- **Fail-open & precision:** MF-053 (fail closed on corrupt config), MF-027 (anchor the regex — usability, low risk), MF-028 (boolean force-push check), MF-051 (wire or delete `safety.block_destructive_commands`).
- **Acceptance:** a forged `mv` manifest write is blocked; a fabricated SHA is rejected; the git guards fire from a fresh clone with `guards.git_hooks: true`; corrupt config denies rather than allows; a one-task-per-bullet wave still dispatches in parallel; an enabled GitHub connector actually posts a status/PR back to its issue.

### Wave 3 — Review-pipeline reliability & integrity
**Charter:** give `review-gate.md` a real dispatch/persistence contract.
- **Stalls (multi-pronged, retires IMP-046/058/071):** MF-014 — reviewer `maxTurns` raise/removal, defensive `run_in_background: false`, verdict-schema-or-retry, and Agent-Teams async persistence.
- **Granularity unblock:** MF-013 (resolve unit ID before evidence check — unblocks group/feature mode) + MF-015 (emit per-(reviewer×task) with a `review_unit` field; "coverage missing" ≠ "no violation") + MF-016 (validate `emit-event` numeric fields).
- **Contract hardening:** MF-058 (one canonical verdict-file name + test) + MF-059 (trust-boundary language: only the dispatch prompt is authoritative) + MF-017/MF-019 (rename `resolved`, wire-or-remove `require_all_approve`) + MF-020 (pipeline de-sprawl).
- **Acceptance:** a group-granularity task reaches DONE; a stalled reviewer retries deterministically; every gate persists the same filename; an injected "soften your verdict" message is contractually ignored.

### Wave 4 — Docs/config sync, HITL, heartbeat, hygiene
**Charter:** add forcing functions so drift can't silently return; clear the low/cosmetic tail.
- **Docs/config:** MF-049 (regenerate the Execution Engine section), MF-048 (`.bak` pruning + exclude/rotate secret-bearing backups), MF-050 (migrate from `post-compact.sh`), MF-029 (add 5 hooks to CLAUDE.md), MF-062 (post-loop retirement step + mark IMP-047/048/056/072 retired).
- **HITL integrity:** MF-006 (mechanical HITL gate on the default sequential dispatch path — the operator-reported "proceeded anyway" incident).
- **Heartbeat:** MF-039 (atomic `mkdir` lock), MF-044 (surface `start_command_failed`).
- **Cosmetic/low tail:** MF-007, MF-008, MF-012 (resolve the PLAUSIBLE hedge via a live check), MF-018, MF-030, MF-031, MF-032, MF-033 (shared tokenizer), MF-037, MF-041/MF-054/MF-056 (guard-branch tests), MF-042 (resolve via deliberate dispatch), MF-043, MF-045, MF-046, MF-057.
- **Process gaps (not code findings):** a local-mode-aware verification mechanism for read-only audit tasks (the qa "zero files written is unverifiable" concern); record the audit-methodology caveat in the objective's history.

### improvements.md consolidation
The 74-item backlog (IMP-001…074, all `open`) is inventoried in full in `dimension-8-findings.md` (lines 177–268). Rather than a 74-row table here, disposition is by category (auditable against that source):
- **~40 items are review-board telemetry** ("Review rejection: TASK-X/reviewer", "N retry attempts") — historical bookkeeping, **not standalone defects**; no code fix. Recommend closing en masse once **MF-062**'s retirement mechanism ships (Wave 4).
- **IMP-046/058/071 → MF-014 (Wave 3)** — the pre-existing backlog records of the reviewer-stall defect; fold into MF-014's fix, don't treat as separate items.
- **IMP-047/048/056/072 → retire** — already fixed at the source level (dim-8 RT-02/RT-03 confirmed); mark retired-by-fix.
- **The remaining defect-bearing items** map to the cluster/wave of their subject finding (each is a re-report of a defect already in the MF-register).
- The closure mechanism itself is MF-062 (Wave 4) — until it ships, this mapping goes stale on the next objective. **A full per-item mapping table is deferred to the MF-062 work item** rather than duplicated (and drifted) here.
