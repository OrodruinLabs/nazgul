# Per-Stage Model Routing

**Date:** 2026-04-14
**Status:** Approved

## Summary

Allow users to configure which AI model (Opus, Sonnet, Haiku) runs each pipeline stage. Ships with balanced defaults, configurable via `/nazgul:config` interactive prompts or direct config editing.

## Motivation

Different pipeline stages have different complexity needs:
- Planning requires deep reasoning (Opus)
- Implementation is code generation (Sonnet)
- Post-loop tasks are mechanical (Haiku)

Letting users configure this optimizes cost, speed, and quality per their priorities.

## Design

### Config Schema

`nazgul/config.json` gains a `models` section (already partially exists in templates):

```json
"models": {
  "planning": "opus",
  "discovery": "sonnet",
  "docs": "sonnet",
  "review": "sonnet",
  "implementation": "sonnet",
  "specialists": "sonnet",
  "post_loop": "haiku",
  "default": "sonnet"
}
```

Valid values: `opus`, `sonnet`, `haiku`, `inherit`.

### Agent-to-Stage Mapping

| Config key | Agents |
|-----------|--------|
| planning | planner |
| discovery | discovery |
| docs | doc-generator |
| review | review-gate, feedback-aggregator, all reviewer agents |
| implementation | implementer |
| specialists | frontend-dev, mobile-dev, devops, cicd, db-migration, designer |
| post_loop | documentation, release-manager, observability |
| default | fallback for any agent not explicitly mapped |

### Model Routing Mechanism

**Layer 1 — Agent tool dispatch (primary):** When the start skill or any orchestrator spawns an agent via `Agent()`, it reads `nazgul/config.json` models section and passes the `model` parameter. The Agent tool's `model` parameter takes precedence over agent frontmatter, so this works with read-only marketplace installs.

**Layer 2 — Generated reviewer frontmatter (secondary):** Project-level reviewers generated by `/nazgul:init` or `/nazgul:discover` are written to the project directory (writable). These get `model:` in their frontmatter based on `models.review` config value. This makes them self-documenting and functional even if spawned outside the main pipeline.

**No plugin files are modified. Config is the single source of truth.**

### Presets

| Preset | Planning | Discovery | Docs | Review | Implementation | Specialists | Post-loop |
|--------|----------|-----------|------|--------|----------------|-------------|-----------|
| Balanced (default) | opus | sonnet | sonnet | sonnet | sonnet | sonnet | haiku |
| Quality | opus | opus | opus | opus | opus | opus | opus |
| Fast/cheap | sonnet | haiku | haiku | haiku | sonnet | sonnet | haiku |

### New Skill: `/nazgul:config`

**Purpose:** View and change Nazgul settings after init without re-running the full init pipeline.

**Allowed tools:** Read, Write, Edit, Bash, Glob, Grep, AskUserQuestion

**Flow:**

1. Read `nazgul/config.json` and display current settings:
   ```
   Nazgul Configuration
   ═══════════════════════════════════════

   Models:
     Planning:        opus
     Discovery:       sonnet
     Docs:            sonnet
     Review:          sonnet
     Implementation:  sonnet
     Specialists:     sonnet
     Post-loop:       haiku
     Default:         sonnet

   Formatter:         disabled
   Notifications:     say 'Nazgul loop complete'
   ```

2. `AskUserQuestion` (multiSelect: true):
   - header: "Settings"
   - question: "What would you like to change?"
   - options: "Model assignments", "Formatter", "Notifications"

3. Based on selection, run the relevant sub-flow.

### Model Assignment Sub-flow

1. `AskUserQuestion`:
   - header: "Models"
   - question: "How would you like to configure models?"
   - options:
     - "Use a preset" — "Quick setup with balanced defaults"
     - "Customize per stage" — "Pick a model for each pipeline stage"

2. **If preset:** present preset picker (Balanced, Quality, Fast/cheap)

3. **If per-stage:** present `AskUserQuestion` calls batched 4 stages at a time (tool limit is 4 questions per call). First batch: planning, discovery, docs, review. Second batch: implementation, specialists, post-loop. Each question offers Opus/Sonnet/Haiku options.

4. Update `nazgul/config.json` with new `models` section

5. Rewrite `model:` field in project-level generated reviewer agents (in `nazgul/agents/` or `.claude/agents/generated/`)

### Init Integration

During `/nazgul:init` Step 7 (Optional Features), add a model configuration step:

1. Show the balanced defaults
2. `AskUserQuestion`:
   - header: "Models"
   - question: "Customize model assignments per pipeline stage?"
   - options:
     - "Use defaults (Recommended)" — "Opus for planning, Sonnet for implementation, Haiku for post-loop"
     - "Customize" — "Choose a model for each pipeline stage"
3. If customize: run the same model assignment sub-flow as `/nazgul:config`
4. If defaults: write the balanced preset to config

### Start Skill Changes

The start skill and all orchestrator agents must read `nazgul/config.json` models section before spawning each agent and pass the `model` parameter:

```
model_for_stage = config.models[stage] || config.models.default || "sonnet"
Agent({ subagent_type: "nazgul:planner", model: model_for_stage, ... })
```

Affected files:
- `skills/start/SKILL.md` — all agent dispatches
- `agents/review-gate.md` — reviewer dispatches
- `agents/team-orchestrator.md` — parallel team dispatches
- `agents/implementer.md` — specialist dispatches

### What Changes When Models Are Updated via `/nazgul:config`

1. `nazgul/config.json` updated with new model values
2. Generated reviewer agent frontmatter rewritten with new `model:` field
3. Next pipeline run automatically picks up new models (config is read at runtime)
