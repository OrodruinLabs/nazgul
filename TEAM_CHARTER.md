# Hydra Team Charter

Defines the roles, responsibilities, communication protocols, and coordination model for all agents in the Hydra Framework.

---

## The Team

Hydra's agent roster is conditional — Discovery instantiates only the agents the project needs. This charter covers all 29 possible agents, grouped by function.

### Pipeline Agents (Always Active)

These agents form the core loop and are present in every Hydra run.

| Agent | Role | Input | Output | Authority |
|-------|------|-------|--------|-----------|
| **Discovery** | Classifies project FIRST (5 types: GREENFIELD/BROWNFIELD/REFACTOR/BUGFIX/MIGRATION), scans codebase, generates ONLY relevant agents (not all 29), must cite evidence (file + line) for every detection | Codebase files, git history | `hydra/context/`, `hydra/config.json`, `.claude/agents/generated/` | Unilateral: classification, agent selection |
| **Doc Generator** | Produces project documents based on classification | Context files, classification, objective | `hydra/docs/` | Unilateral: document content |
| **Planner** | Decomposes objective into dependency-ordered tasks | Documents, context, objective | `hydra/plan.md`, `hydra/tasks/TASK-*.md` | Unilateral: task structure, dependencies, priorities |
| **Implementer** | Builds each task, delegates to specialists | Task manifest, context, pattern references | Modified source files, updated task manifests | Unilateral: implementation approach. Delegates to specialists. Must stay within the task's `file_scope` — files outside scope require task manifest update first. |
| **Review Gate** | Orchestrates the review board, manages state transitions | Implemented task, reviewer verdicts | Review verdicts, state transitions | Consensus: requires all reviewers to approve |
| **Feedback Aggregator** | Consolidates rejection feedback into actionable items | Individual reviewer feedback files | `hydra/reviews/TASK-NNN/consolidated-feedback.md` | Unilateral: feedback organization |
| **Team Orchestrator** | Manages Agent Teams for parallel execution | Config parallelism settings, task dependencies | Team coordination, parallel task assignment | Unilateral: parallelization decisions |

### Reviewer Agents (Conditional)

Generated by Discovery based on project needs. All reviewers share the same interface.

| Agent | Specialization | Spawned When |
|-------|---------------|-------------|
| **Architect Reviewer** | System design, patterns, coupling | Always (core reviewer) |
| **Code Reviewer** | Code quality, naming, complexity, DRY | Always (core reviewer) |
| **Security Reviewer** | Vulnerabilities, injection, auth flaws | Always (core reviewer) |
| **QA Reviewer** | Test coverage, edge cases, test quality | Project has test suite |
| **Performance Reviewer** | N+1 queries, memory leaks, algorithmic complexity | Backend or data-heavy projects |
| **A11y Reviewer** | WCAG compliance, ARIA, keyboard navigation | Frontend or UI projects |
| **DB Reviewer** | Schema design, migrations, indexes | Projects with database |
| **API Reviewer** | REST/GraphQL design, versioning, error contracts | Projects exposing APIs |
| **Type Reviewer** | Type safety, generics, inference | TypeScript, Flow, or typed language projects |
| **Infra Reviewer** | Docker, K8s, cloud config, IaC | Projects with infrastructure code |
| **Dependency Reviewer** | Package versions, vulnerability audit, license compliance | Projects with package managers |
| **Mobile Reviewer** | Platform guidelines, performance, battery usage | Mobile projects |
| **Data Reviewer** | Data integrity, privacy, GDPR, data flow | Data-intensive projects |

**Reviewer Contract**: Every reviewer must:
1. Read only — never modify project files
2. Return a structured verdict: APPROVE, REJECT, or CONCERN
3. Include a confidence score (0-100) with every finding
4. Write their review to `hydra/reviews/TASK-NNN/[reviewer-name].md`
5. Structure each finding with required fields: severity (HIGH/MEDIUM/LOW), confidence (0-100), file path, category (security/performance/correctness/style/etc.), verdict (APPROVE/REJECT/CONCERN), issue description, and fix suggestion. Include a pattern reference when relevant.

### Specialist Agents (Conditional)

Delegated to by the Implementer for domain-specific work.

| Agent | Specialization | Delegated When |
|-------|---------------|---------------|
| **Designer** | Design system, visual direction, component specs | UI tasks on projects with frontend |
| **Frontend Dev** | React/Vue/Svelte component implementation | Frontend implementation tasks |
| **Mobile Dev** | iOS/Android/React Native implementation | Mobile platform tasks |
| **DevOps** | Docker, K8s, cloud configs, IaC | Infrastructure tasks |
| **CI/CD** | Pipeline generation, workflow configs | Build/deploy automation tasks |
| **DB Migration** | Safe schema changes, migration scripts | Database schema tasks |

**Specialist Contract**: Specialists:
1. Receive a delegation brief at `hydra/tasks/TASK-NNN-delegation.md`
2. Implement within the scope defined in the brief
3. Return control to the Implementer when complete
4. Do not modify files outside their delegation scope

#### Specialist Coordination Details

**Designer → Frontend Dev Pipeline**: The Designer outputs a design system specification (typography, color, spacing, component library) and writes it to `hydra/docs/design-system.md` and `hydra/docs/design-tokens.json`. The Frontend Dev consumes these specs when implementing UI components. The Designer runs first; Frontend Dev runs after.

**DevOps → CI/CD Pipeline**: DevOps produces infrastructure configurations (Dockerfiles, K8s manifests, cloud configs). CI/CD consumes these to generate pipeline configurations (GitHub Actions, GitLab CI, etc.). DevOps runs first if both are needed.

**DB Migration Safety Protocols**: The DB Migration specialist follows the expand-contract pattern for schema changes:
1. **Expand**: Add new columns/tables without removing old ones
2. **Migrate**: Move data from old to new structure
3. **Contract**: Remove old columns/tables only after verification
All migrations must include rollback scripts. Zero-downtime migrations are the default approach.

**Mobile Dev Platforms**: Handles iOS (Swift/SwiftUI), Android (Kotlin/Jetpack Compose), and cross-platform (React Native/Flutter). Platform-specific delegation briefs specify target platform constraints.

### Post-Loop Agents (Conditional)

Run after all tasks are DONE, before HYDRA_COMPLETE.

| Agent | Role | Runs When |
|-------|------|----------|
| **Documentation** | Updates README, API docs, changelog | Configured in `agents.post_loop` |
| **Release Manager** | Versioning, release notes, tag creation | Configured in `agents.post_loop` |
| **Observability** | Verifies logging, metrics, error tracking | Configured in `agents.post_loop` |

#### Execution Order and Rules

Post-loop agents run in strict sequence after ALL tasks reach DONE:

1. **Documentation** → Updates README, API docs, changelog based on all implemented tasks
2. **Release Manager** → Handles version bumps, release notes, git tags
3. **Observability** → Verifies logging, metrics, error tracking are properly configured

**Failure handling**: If any post-loop agent fails, HYDRA_COMPLETE does not fire. The failure is logged and the user is notified. The loop can resume with `/hydra-start` to retry.

**Authority scope**: Post-loop agents may modify documentation, release artifacts, and observability configs. They must NOT modify application source code or test files.

## Communication Protocol

Agents communicate exclusively through files. There is no direct agent-to-agent messaging within the Hydra pipeline.

### File-Based Communication

| From | To | Medium | Example |
|------|----|--------|---------|
| Discovery | All downstream | `hydra/context/`, `hydra/config.json` | Project profile, classification, reviewer list |
| Doc Generator | Planner | `hydra/docs/` | PRD, TRD, ADRs |
| Planner | Implementer | `hydra/tasks/TASK-*.md`, `hydra/plan.md` | Task manifests with acceptance criteria |
| Implementer | Specialists | `hydra/tasks/TASK-NNN-delegation.md` | Delegation briefs |
| Implementer | Review Gate | Task manifest status change to IMPLEMENTED | State transition signals readiness |
| Each Reviewer | Feedback Aggregator | `hydra/reviews/TASK-NNN/[name].md` | Individual review verdicts |
| Feedback Aggregator | Implementer | `hydra/reviews/TASK-NNN/consolidated-feedback.md` | Actionable feedback |
| Stop Hook | All agents (next iteration) | stderr message, checkpoint | Loop state, continue instructions |
| Any agent | External tools | `hydra/notifications.jsonl` | Events for OpenClaw or monitoring |

### The Recovery Pointer as Coordination Signal

The Recovery Pointer in `plan.md` acts as the coordination bus. Every agent reads it first. Every agent that changes state updates it. This is how agents "hand off" work across iterations:

```
Implementer finishes → updates Recovery Pointer to "Review Gate should pick up TASK-004"
Review Gate reads → sees TASK-004 is IMPLEMENTED → runs review pipeline
Review Gate finishes → updates Recovery Pointer to "Implementer should read feedback for TASK-004"
```

### Git Log as Audit Trail

In AFK mode, every state transition generates a commit:

```
hydra: TASK-001 → IN_PROGRESS (claimed)
hydra: TASK-001 → IMPLEMENTED (tests: 5/5, lint: clean)
hydra: TASK-001 → DONE (all reviewers approved)
hydra: TASK-004 → CHANGES_REQUESTED (retry 1/3, code-reviewer rejected)
hydra: TASK-008 → BLOCKED (requires API keys)
```

`git log --oneline --grep="hydra:"` reconstructs the entire project history.

### Notifications for External Integration

The stop hook writes structured events to `hydra/notifications.jsonl` for external tools:

| Event Type | When Emitted | Payload |
|-----------|-------------|---------|
| `task_complete` | A task transitions to DONE | task ID, completion count, total |
| `blocked` | A task transitions to BLOCKED | task ID, blocked reason, requires_human flag |
| `loop_complete` | All tasks DONE + post-loop complete | total tasks, branch, summary |
| `security_rejection` | Task blocked for security reasons | task ID, security reason (fires ALWAYS, ignores notifications.enabled) |
| `git_conflict` | Unmerged files detected | task ID, conflict details |
| `post_loop_failure` | Post-loop agent fails | agent name, failure details |

**Security notifications are unconditional.** The `security_rejection` event fires regardless of the `notifications.enabled` setting. This ensures security issues are never silently swallowed.

External tools like OpenClaw can `tail -f hydra/notifications.jsonl` to monitor loop progress and react to events.

## Delegation Briefs

When the Implementer delegates work to a specialist, the delegation is formalized in a brief file.

### Format

Delegation briefs are written to `hydra/tasks/[TASK-ID]-delegation.md` with this structure:

```
# Delegation Brief: TASK-NNN

## Delegated To
[Specialist agent name]

## Task Context
[What the broader task is about, relevant background]

## Specific Scope
[Exactly what the specialist should implement — files to create/modify, features to build]

## Expected Output
[What the specialist produces — files, configs, specs]

## Constraints
- Technology: [specific frameworks, libraries, versions]
- Patterns: [reference existing patterns to follow]
- Performance: [any perf requirements]
- Accessibility: [any a11y requirements]

## Return Protocol
When complete, update the task manifest and return control to the Implementer.
```

### Rules

- Written by the Implementer, consumed by the Specialist
- Specialist stays within the brief's scope — no modifications outside it
- Implementer reviews specialist output before marking task as IMPLEMENTED
- If the brief needs revision, the Implementer writes an updated brief (same file, appended)

## Coordination Model

### Sequential Pipeline (Default)

```
Discovery → Doc Generator → Planner → [Implementer → Review Gate]* → Post-Loop
                                       └── loops per task ──────┘
```

One task at a time. The stop hook manages iteration between Implementer and Review Gate.

### Parallel Pipeline (With Agent Teams)

When `parallelism.enabled` and `parallelism.use_agent_teams` are true:

1. **Independent tasks** (no shared dependencies) can be implemented in parallel by the Team Orchestrator
2. **Reviews** can run in parallel (all reviewers evaluate simultaneously)
3. **Maximum parallel teammates** is configurable (`parallelism.max_parallel_teammates`, default: 4)

Parallel execution still respects:
- Dependency ordering (a task cannot start until its dependencies are DONE)
- The review gate (all reviewers must still unanimously approve)
- State machine transitions (no state skipping)

#### Team Orchestrator Responsibilities

The Team Orchestrator manages all parallel coordination:

1. **Task group identification**: Analyzes task dependencies and file scopes to find independent groups
2. **File scope validation**: Verifies parallel implementers have ZERO overlapping files. If overlap is detected, those tasks run sequentially instead.
3. **Team creation**: Spawns Agent Teams with appropriate size (up to `max_parallel_teammates`)
4. **Progress monitoring**: Tracks parallel task completion and unblocks dependent tasks
5. **Fallback**: If Agent Teams are unavailable (env var not set), all work runs sequentially

#### Parallel Reviews

When `parallelism.parallel_reviews` is true, ALL reviewers run as simultaneous teammates in an Agent Team. Each reviewer:
- Reads the changed files independently
- Writes their review to `hydra/reviews/TASK-NNN/[reviewer-name].md`
- Returns their verdict

The Review Gate waits for ALL reviewer teammates to complete before determining the final verdict.

### Handoff Protocol

When one agent completes its phase and another must take over:

1. **Completing agent** updates the task manifest with new status
2. **Completing agent** updates the Recovery Pointer in `plan.md`
3. **Completing agent** commits changes (AFK mode)
4. **Stop hook** fires, writes checkpoint, outputs continue message
5. **Next agent** reads Recovery Pointer, checkpoint, and task manifest
6. **Next agent** resumes from the documented Next Action

There is no direct invocation between pipeline stages. The stop hook is the scheduler.

## Agent Lifecycle

### Instantiation

1. Discovery scans the project and determines which agents are needed
2. Discovery writes agent configurations to `hydra/config.json` under `agents.*`
3. Discovery generates project-tailored reviewer agents in `.claude/agents/generated/`
4. All subsequent pipeline stages read the agent list from config

### During Loop

- Pipeline agents are invoked by the stop hook's continue message or by `/hydra-start`
- Reviewers are invoked by the Review Gate (sequentially or in parallel)
- Specialists are invoked by the Implementer via delegation briefs
- Post-loop agents are invoked by the Review Gate after all tasks are DONE

### Termination

- The loop ends when all tasks are DONE and post-loop is complete (HYDRA_COMPLETE)
- Or when an exit condition is met (max iterations, consecutive failures, AFK timeout, pause flag)
- Agent state does not persist between loops — each new `/hydra-start` reads fresh from disk

## Context Isolation

Skills that perform read-only operations run with `context: fork` to prevent polluting the main session's context window. This means they execute in an isolated context that is discarded after the skill completes.

**Forked skills** (isolated context):
- `/hydra-status`, `/hydra-task`, `/hydra-pause`, `/hydra-log`, `/hydra-reset`
- `/hydra-review`, `/hydra-discover`, `/hydra-context`, `/hydra-simplify`, `/hydra-docs`

**Non-forked skills** (share main context):
- `/hydra-init` (needs to modify main session state)
- `/hydra-start` (drives the main loop)

Forked skills can safely read any file without worrying about context budget. Their results are presented to the user but their file reads don't consume main session context.

## Responsibilities Matrix

| Responsibility | Primary | Backup |
|---------------|---------|--------|
| Project understanding | Discovery | Human (via /hydra-discover) |
| Requirements documentation | Doc Generator | Human (manual docs) |
| Task decomposition | Planner | Human (via /hydra-task add) |
| Code implementation | Implementer + Specialists | Human (manual coding) |
| Code review | Review Board | Human (via /hydra-review) |
| Feedback consolidation | Feedback Aggregator | Human (reading review files) |
| State management | Stop Hook + Pre-Compact Hook | Human (via /hydra-status, /hydra-task) |
| Recovery | Session-Context Hook + Checkpoints | Human (via /hydra-start --continue) |
| Documentation | Documentation Agent (post-loop) | Human (via /hydra-docs) |
| Release management | Release Manager (post-loop) | Human (manual release) |
| Observability | Observability Agent (post-loop) | Human (manual setup) |

For every automated responsibility, there is a manual fallback. Hydra augments human capability; it does not replace human authority.
